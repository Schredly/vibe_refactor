Prompt 1 — App scaffold + core decisions

Build a full-stack web MVP called Legacy BIO to track biomarker assay samples from intake → processing → approval.
Tech stack requirements:

Backend: Node.js + Express

Database: PostgreSQL

Frontend: React

UI: Material UI (or a minimal component library), responsive for desktop/laptop (no mobile app features)

Use REST APIs (no GraphQL)

Core product requirements:

Track Sample lifecycle with clear statuses and timestamps

Role-based access control with roles: technician, approver, manager

Auditability first: every meaningful change creates an immutable audit record (who/what/when)

Moderate volume performance: pagination + server-side filtering (no heavy optimization)

Create a monorepo structure:

/server for Express

/client for React

Shared env config and a clean README with setup instructions

Include database migrations/seed scripts

Deliverables:

Working login + session handling

DB schema + migrations

Basic pages scaffolded: Login, Status Board, Sample Detail, Manager Dashboard

Placeholder “API stub layer” for future LIMS integration (but do NOT integrate external systems)

Prompt 3 — Authentication + RBAC

Implement simple authentication (no SSO):

Login with username + password

Password hashing (bcrypt)

Session-based auth (cookie session) OR JWT stored securely; choose one and implement correctly

Middleware: requireAuth and requireRole(roles[])

RBAC rules:

Technician:

Create sample

Update status (except cannot set COMPLETED directly)

Edit processing_notes

View sample detail + audit trail

Approver:

View all samples

Edit result_summary

Approve result (sets status to COMPLETED, sets approved_by + approved_at)

Manager:

View all samples + dashboard metrics + export CSV

View sample detail + audit trail

(Optional) can assign technician to sample

UX requirement:

Login screen should route user to the correct landing page:

Technician/Approver → Status Board

Manager → Dashboard (with nav access to Status Board)

Also log security events into audit_log:

Successful login (LOGIN_SUCCESS)

Failed login attempt (LOGIN_FAILED) with username and timestamp (don’t store raw password)

Prompt 4 — Audit logging engine (core system behavior)

Build a reusable audit logging utility in the backend:

A single function like logAudit({entityType, entityId, action, userId, changedFields, notes})

For changedFields, store a JSON diff in a consistent format:

{ fieldName: { before: <value>, after: <value> }, ... }

Ensure audit_log is append-only: no updates/deletes (enforce in code; optional DB rule)

Trigger audit records for:

Sample created

Status changed

Sample fields edited (processing_notes, result_summary, assigned technician)

Approval action

CSV export run (EXPORT_CSV) with filters used

Manager dashboard viewed (optional: VIEW_DASHBOARD)

Also write status changes to status_history:

On every status change, create a row capturing from/to/user/timestamp/comment

Return audit trail in chronological order (newest last) for a sample.

Prompt 5 — Backend APIs (REST) with filters + pagination

Implement REST endpoints with validation and role enforcement:

Auth:

POST /api/auth/login

POST /api/auth/logout

GET /api/auth/me

Samples:

POST /api/samples (technician)

GET /api/samples (all roles) with server-side filters + pagination:

filters: sampleId (partial), status, assayType, receivedFrom, receivedTo, assignedTechnicianId

pagination: page, pageSize

sort: default updated_at desc

GET /api/samples/:id

PATCH /api/samples/:id (role-based field editing)

POST /api/samples/:id/status to change status with {toStatus, comment}

Approval:

POST /api/samples/:id/approve (approver only)

Valid only if current status is AWAITING_APPROVAL

Writes audit + status history + sets approved fields

Audit:

GET /api/samples/:id/audit returns audit trail

GET /api/samples/:id/history returns status history

Metrics & dashboard:

GET /api/dashboard/summary (manager only)

counts by status given optional filters: date range, assayType, technician

GET /api/dashboard/turnaround (manager only)

average turnaround time: avg(completed_at - received_at) for completed samples in range

GET /api/dashboard/delayed (manager only)

“delayed samples” defined as samples not completed and in same status longer than X hours (configurable, default 24)

GET /api/dashboard/export.csv (manager only)

generates CSV of sample rows + computed turnaround time where applicable, includes filter metadata in header comments

API design requirements:

Use consistent error responses

Validate payloads (zod/joi) and sanitize inputs

Avoid N+1 queries; do joins where needed

Make sure pagination returns {items, page, pageSize, total}
